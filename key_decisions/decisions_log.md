| 17 | 3.0 | Calendar integration scope | (A) Google Calendar only · (B) Google + Outlook from launch · (C) Abstracted integration layer (both, same code path) | A = fastest to ship, covers current user · B = large scope increase, different OAuth entirely · C = right architecture long-term but premature · Google-only users: no regression. | 18 | 3.0 | Committed day primary surface | (A) Slack brief only · (B) In-app Today view only · (C) Both — Slack as ambient pre-game, in-app as decision surface | A = no decision surface, planning stays implicit · B = misses ambient reach before app is opened · C = each channel does one job well. | 19 | 3.3 | Pattern Memory readiness gate | (A) Time-based (e.g. after 4 weeks) · (B) Global outcome count floor · (C) Per-category floor · (D) Combined global + per-category | A = punishes engagement, power user and casual user get same gate regardless of data · B = global floor alone risks surfacing patterns from too-small category samples · C = per-category alone risks never showing anything if user has diverse outcomes · D = global floor (20) ensures minimum corpus, per-category floor (8) ensures specific patterns are meaningful | **D — 20 closed outcomes global floor before any pattern surfaces. 8 data points per category before that category's pattern surfaces. | 20 | 3.x | Library vs Pattern Memory ordering | (A) Ship together in one phase · (B) Library first (3.2), Pattern Memory second (3.3) | A = two hard things at once, neither done cleanly · B = Library is retrieval (different architecture from inference), Library feeds Pattern Memory as a data source — shipping Library first means Pattern Memory has a proper searchable corpus to work from when it arrives. | **B — Library is Phase 3.2, Pattern Memory is Phase 3.3. Architecturally separate. Library ships first and becomes a data source for Pattern Memory.** | Feb 20 |
